Most EF Core performance bugs are invisible until you look at the SQL. Slow queries, missing filters, N+1 problems - they all show up the moment you start logging what EF Core actually sends to the database.
There are two practical approaches for development, and each one belongs in a different context.

🔹 Global logging via LogTo

The simplest way to get SQL output. You configure it directly in AddDbContext using LogTo. It pipes EF Core log output to any Action<string> - console, debug output, a file. Combine it with EnableSensitiveDataLogging() and you also see the actual parameter values, not just placeholders. This is the baseline setup every project should have in development.

🔹 ToQueryString() for targeted inspection

This one is different - it is not logging infrastructure, it is a debugging tool. Call ToQueryString() on any IQueryable<T> before materialisation and you get the exact SQL string EF Core would generate. No need to run the query. No need to scan log output. You see the SQL right there in your debugger or console. Invaluable when you are investigating a specific query and do not want noise from the rest of the application.

🔹 When does this matter
Global logging catches problems you did not know existed. ToQueryString() confirms what you suspect. Using both in the right context gives you full visibility into what EF Core is doing without any guesswork.

✦ Use LogTo with Console.WriteLine or Debug.WriteLine for quick local development setup

✦ Use EnableSensitiveDataLogging() only in development - it logs parameter values which includes user data

✦ Never leave EnableSensitiveDataLogging() enabled in production - treat it the same as hardcoded credentials

✦ Use ToQueryString() when debugging a specific query - it is faster than scanning log output


If you want to use EF Core logging on your main branch in development you can consider to add app.IsDevelopment() 

check when registering the DbContext.


My default setup on every project is LogTo wired to console output in development.

I only reach for ToQueryString() when I am investigating a specific query that looks wrong.
